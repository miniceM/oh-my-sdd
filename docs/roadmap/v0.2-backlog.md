## 🔴 P0：v0.1 实测发现的核心 bug（阻塞真正可用）

### B-1 baseline 注入实测不工作 ✅ 已解决（commit `ad1e5c9`）
**根因**：Claude Code plugin SessionStart hook 的 `additionalContext` 有已知 bug #16538（silently dropped）。我们的 hook 跑了、输出对了、JSON 也合法，但 Claude Code 不把它放进 system prompt。

**走错的弯路**：
- 误以为 `sessionStart.skill` 字段会自动加载 skill 内容 → 错，skill 只在被 Skill 工具调用时激活
- 误以为 hook 输出有问题 → 错，hook 输出完美

**最终方案**：install.js 写入 `~/.claude/CLAUDE.md`（用户级，每次会话自动加载），用 BEGIN/END marker 段实现幂等升级与卸载清理。

**遗留**：
- session-start.js 仍输出 additionalContext（被 #16538 静默 drop），算 defense-in-depth，等 Anthropic 修 bug 后自动恢复工作
- 监控 [issue #16538](https://github.com/anthropics/claude-code/issues/16538) 状态，修复后可以删除 CLAUDE.md 注入路径回归"纯 plugin"方案

### B-2 npm postinstall 输出默认被吞 ✅ 已缓解
**现象**：`npm install -g @cli-tools/oh-my-sdd` 不加 `--foreground-scripts` 完全静默。
**现状**：已在 README 推荐 `--foreground-scripts` + `npm config set foreground-scripts true`。
**v0.2 选项**：
- 接受现状（行业惯例）
- 在 npm registry 描述里加显式提示
- IT 团队统一配 `foreground-scripts=true`

## 🟡 P1：官方对齐与功能补全

### F-1 commands/ 迁移到 skills/ ✅ 完成（alpha 测试发现）
**触发**：用户在 demo 工程实测发现 `/oh-my-sdd:sdd-spec` 等 5 个命令带 plugin 前缀，与 `/security-check` 等无前缀 skills 不一致——用户体验割裂。

**Claude Code 命名规则**：
- `commands/*.md` → `/<plugin>:<command>`（带前缀）
- `skills/<name>/SKILL.md` → `/<skill-name>`（无前缀）

**已完成**（commit 待补 hash）：
- 5 个命令从 `commands/` 迁到 `skills/<name>/SKILL.md`
- 每个 SKILL.md frontmatter 加 `name: sdd-X`
- plugin.json 移除 `commands` 字段
- package.json `files` 数组移除 `commands/`

**经验**：F-1 在 backlog 里本是"官方推荐方向"（nice-to-have），alpha 测试升级为 must-have。**证明 alpha 阶段价值——5 分钟用户测试比 1 小时静态审查更有效。**

### F-2 用 userConfig 替代手改 config.json
**原因**：官方 plugin 提供 `userConfig` 字段，用户在 `/plugin` 里交互配置，不用手改 `~/.oh-my-sdd/config.json`。
**改造**：
```json
"userConfig": {
  "dop_endpoint": { "type": "string", "title": "DOP endpoint URL", "required": true },
  "aih_system_name": { "type": "string", "title": "AIH system 名", "default": "sdd" },
  "telemetry_disabled": { "type": "boolean", "title": "禁用遥测", "default": false }
}
```
hook 里读 `${user_config.dop_endpoint}` 替代 `loadConfig()`。敏感字段（如 DOP 鉴权）用 `"sensitive": true` 走 keychain。

### F-3 files_touched 真正上报到 session.end
**现状**：PostToolUse 增量记录 files_touched，但 session.end 事件 schema 没这个字段（spec § 7.1）。记录了不用。
**改造**：
- session.end schema 加 `files_touched: [{path, count}]`（或聚合统计）
- session-end.js 读 meta.files_touched 加到 event
- DOP 后端配套改

### F-4 slash_commands 去重
**现状**：用户跑 5 次 `/sdd-plan`，session.end 收到 `slash_commands_used: ['sdd-plan','sdd-plan','sdd-plan','sdd-plan','sdd-plan']`。
**改造**：session-end.js 读 meta.slash_commands 时去重 + 计数：
```js
const counts = meta.slash_commands.reduce((acc, c) => ({...acc, [c]: (acc[c]||0)+1}), {});
// 上报 slash_commands_used: { sdd-plan: 5 }
```

### F-5 SessionEnd 不可靠场景补传 session.end
**现状**：用户关闭终端 → SessionEnd 不触发 → session.start 上报了但 session.end 没上报 → DOP 后端有孤儿 session.start。
**改造**：
- session-start.js 检测孤儿 session meta（有 start_sha 但没对应的 session.end 记录）
- 补传一个 session.end（用 PostToolUse 增量记录的 files_touched 推断 code_delta）
- 标 `recovered: true` 字段让 DOP 后端识别

### F-6 checkNodeVersion 单段输入 NaN bug
**位置**：`hooks/lib/platform.js:6-13`
**问题**：`checkNodeVersion('18')` 会 NaN 比较，返回 true（错误）。
**修复**：guard 输入或拆段补 0。
**影响**：install.js preflight 用 `checkNodeVersion('18.0.0')` 是 3 段没问题，但 API 本身有 bug。

## 🚨 P0 战略级缺口：openspec 保鲜机制未落地（v0.3 头号议题）

### S-6 openspec 项目 specs 完全没发挥作用（保鲜承诺落空） ⚠️ 部分修复
**核心问题**：用户引入 openspec 是为了 **spec 资产保鲜**——让 `openspec/specs/` 成为权威的、永远反映系统现状的活文档。原实现度约 25%，承诺基本落空。

**修复进度**（v0.2.1 alpha）：
- ✅ S-6.1 /sdd-spec 加读现状步骤（步骤 4）
- ✅ S-6.2 spec 文件改 delta 格式（ADDED/MODIFIED/REMOVED）
- ✅ S-6.3 /sdd-review 强制 openspec archive（删 mv fallback）+ merge 验证
- ⏳ S-6.4 CI 加 openspec validate（v0.3）
- ⏳ S-6.5 文档化"项目 specs 是保鲜资产"（部分做了——README + 命令明示，待 onboarding 文档）

**遗留**：真演练验证 superpowers 配合 delta 格式输出（risk #1）。

**6 维度评估**（修复后）：
| 维度 | 应该 | 现状 |
|------|------|------|
| 变更工作目录 | 结构化产出 | ✅ 做到 |
| 项目 specs 是权威 | 反映系统现状 | ✅ /sdd-spec/plan 都读 openspec/specs/ |
| Delta 显式化 | 标 ADDED/MODIFIED/REMOVED | ✅ /sdd-spec 强制 delta 格式 |
| 归档时 merge | delta 应用到项目 specs | ✅ 强制 openspec archive，merge 验证 |
| Drift detection | CI 跑 validate | ❌ v0.3 待办（S-6.4） |
| 新变更读现状 | /sdd-spec 先读 openspec/specs/ | ✅ 步骤 4 加读现状 |

**修复方案（5 个子项）**：

#### S-6.1 /sdd-spec 加"读现状"步骤
- 步骤 0：`Read("openspec/specs/<capability>/spec.md")`（如存在）
- 识别本次变更是 ADDED / MODIFIED / REMOVED 哪种
- 与用户确认变更类型

#### S-6.2 spec 文件改用 delta 格式
- 当前：全量重写 spec
- 改为：标注变更类型
  ```markdown
  ## ADDED Requirements
  - <new requirement>
  
  ## MODIFIED Requirements  
  - <old> → <new>
  
  ## REMOVED Requirements
  - <deprecated>
  ```

#### S-6.3 /sdd-review 强制 openspec archive（去掉 mv fallback）
- mv fallback 直接破坏保鲜（无 merge）
- 改为：无 openspec 时阻塞归档，提示用户先装
- 或：写我们自己的 minimal merge 脚本（v0.4 可考虑）

#### S-6.4 CI 加 openspec validate
- `.github/workflows/ci.yml` 加步骤：`openspec validate --all --strict`
- 检测 spec 与代码 drift
- 项目级 specs 过期 → CI 红

#### S-6.5 文档：项目 specs 是保鲜资产
- README 加"openspec/specs/ 是项目权威 specs，反映系统现状"
- /sdd-spec 命令开头明示"先读 openspec/specs/ 再写 delta"
- 写 onboarding 文档：新人怎么读 openspec/specs/ 了解系统

**估时**：S-6.1 + S-6.2 = 1 天；S-6.3 = 半天；S-6.4 = 1 小时；S-6.5 = 2 小时。**合计约 2 天**。

**为什么 P0**：没有这个，整个 plugin 的"spec 资产保鲜"承诺是空话。企业用户用一段时间后会发现"openspec/specs/ 永远是空的，每次变更都从头写"，失去信任。
