# 业务建模 Skill

新项目初始化、微服务拆分、业务模块划分时加载——错误的领域边界会导致后续重构成本剧增。

## 加载决策

| 任务 | Read 这个 resource |
|------|------------------|
| DDD 实践、聚合根设计、领域事件、限界上下文、上下文映射 | `resources/domain-modeling.md` |

## 核心规则（无需 Read resources 也必须遵守）

- **聚合根边界 = 事务边界**——跨越聚合根的事务会破坏一致性
- **微服务划分 = 限界上下文**——按业务能力划分，而非技术分层
- **外部对象仅能通过聚合根 ID 引用聚合内实体**——禁止跨聚合直接引用
- **领域事件命名用过去时态**（`OrderPlaced`）——表示已发生的事实
- **实体有唯一标识，值对象无标识**——区分两者可避免过度设计
- **单个聚合建议 ≤ 5 个实体**——避免聚合过大导致并发冲突

## 何时不应使用

- 业务流程图/BPMN（那是流程建模工具）
- UML 类图（那是图形建模）
- 数据库表设计（→ `db-conventions` skill）
- API 契约设计（→ `api-design` skill）
