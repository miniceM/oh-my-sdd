# 业务模型设计规范

> 定义团队业务领域的建模原则、领域知识沉淀规范和产品知识管理要求。


---

## 1. 领域驱动设计原则

### 1.1 限界上下文

[HARD_RULE] 每个业务模块必须明确定义其限界上下文（Bounded Context）：
- 上下文边界通过包/模块结构显式体现
- 跨上下文通信必须通过防腐层（Anti-Corruption Layer）
- 禁止跨上下文直接引用内部实体

[SOFT_RULE] 限界上下文命名遵循 `业务域.子域` 格式，如 `payment.core`、`user.auth`。

### 1.2 聚合根设计

[HARD_RULE] 聚合根必须满足：
- 保证聚合内数据一致性（事务边界）
- 外部对象仅能通过聚合根 ID 引用聚合内实体
- 聚合根负责验证聚合内所有业务规则

[SOFT_RULE] 单个聚合包含的实体数量不超过 5 个，避免聚合过大导致并发冲突。

### 1.3 值对象

[HARD_RULE] 值对象必须：
- 不可变（所有字段 `final` / 只读）
- 通过值相等性判断（重写 `equals`/`hashCode` 或语言等价物）
- 自验证（构造时校验不变量）

[GUIDELINE] 优先使用值对象而非原始类型表达业务概念（如 `Money` 替代 `BigDecimal`）。

## 2. 领域事件规范

### 2.1 事件命名

[HARD_RULE] 领域事件命名遵循过去时态：`{实体}{动作}ed`，如 `OrderCreated`、`PaymentCompleted`。

### 2.2 事件内容

[HARD_RULE] 领域事件必须包含：
- 事件 ID（UUID）
- 发生时间（UTC ISO 8601）
- 聚合根 ID
- 事件类型（全限定类名）
- 业务数据（仅包含必要字段，不携带完整实体）

[SOFT_RULE] 事件数据序列化使用 JSON Schema 定义，确保向后兼容。

### 2.3 事件发布

[HARD_RULE] 领域事件发布遵循：
- 聚合根内部产生的事件在事务提交后发布
- 禁止在事务内同步调用外部系统
- 使用 Outbox 模式保证事件不丢失

## 3. 产品知识沉淀

### 3.1 业务知识库

[HARD_RULE] 每个业务领域必须维护：
- 核心业务术语表（Ubiquitous Language）
- 业务流程图（关键路径）
- 业务规则清单（可执行或可验证形式）

[SOFT_RULE] 业务知识库与代码同仓维护，存放于 `docs/domain/{业务域}/` 目录。

### 3.2 业务规则表达

[HARD_RULE] 复杂业务规则必须：
- 使用 Given/When/Then 格式描述验收条件
- 对应至少一个自动化测试用例
- 规则变更时同步更新测试用例

示例：
```
Given 用户账户余额为 100 元
When 发起 120 元转账请求
Then 转账失败，返回"余额不足"错误
And 账户余额保持 100 元不变
```

## 4. 领域服务规范

### 4.1 职责边界

[HARD_RULE] 领域服务仅用于：
- 协调多个聚合根的操作
- 调用外部系统获取领域逻辑所需数据
- 执行不属于任何单一聚合根的业务规则

[SOFT_RULE] 领域服务不包含状态，不持有可变字段。

### 4.2 依赖方向

[HARD_RULE] 依赖方向必须遵循：
```
接口层 → 应用层 → 领域层 ← 基础设施层
```
- 领域层不依赖任何外层
- 基础设施层实现领域层定义的接口（依赖倒置）

## 5. 上下文映射

[HARD_RULE] 跨限界上下文的集成模式必须显式声明：

| 模式 | 使用场景 | 要求 |
|------|----------|------|
| 共享内核 | 两个团队共享的通用子域 | 变更需双方评审 |
| 客户-供应商 | 上游/下游关系 | 下游依赖上游公开契约 |
| 防腐层 | 外部系统集成 | 必须转换外部模型为内部模型 |
| 开放主机服务 | 对外提供 API | 必须发布协议语言 |
| 发布语言 | 事件驱动集成 | 事件 Schema 必须版本化 |

[SOFT_RULE] 上下文映射图存放于 `docs/domain/{业务域}/context-map.md`。

## 6. 规格注入指引

[GUIDELINE] 项目级规格引用本规范时：
- 仅注入与当前任务相关的业务上下文
- 业务知识库按功能模块选择性加载
- 上下文映射图仅在涉及跨模块集成时注入
