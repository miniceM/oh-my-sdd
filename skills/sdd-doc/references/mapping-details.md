# 章节映射规则与技术路线（维护者参考）

> 本文档供 skill 维护者了解映射细节与设计演变。**执行 `/sdd-doc` 时无需读取**——脚本已自动处理映射，Claude 只需按 SKILL.md 跑脚本 + 看结果。

## 章节映射规则（v3）

生成文档严格按企业模板章节结构填充：

| spec 产物 | → 模板位置 | 填充方式 |
|-----------|-----------|---------|
| `proposal.md` 背景 | "1 总体说明" → "1.1 项目信息" | proposal 的 `## 背景/目标` 段 |
| `proposal.md` 范围 | "1 总体说明" → "1.2 功能列表" | proposal 的 `## 变更范围` 段 |
| `proposal.md` 验收 | "1 总体说明" → "1.3 验收标准" | proposal 的 `## 验收标准` 段 |
| 每个 `specs/<capability>/` | "2 功能详细分析及设计" → "2.X <capability 显示名>" | deepcopy 完整功能块（含全部表格），改 heading2 标题 |
| capability 的 Requirement | 功能块内**功能概述表**行 1（功能名称）/行 4（功能说明） | 替换示例文字 |
| capability 的 Scenario（WHEN-THEN） | 功能块内**处理逻辑表**（1×1 单元格） | 多行文本 |
| 接口设计/会计核算/影响范围/账务等表 | 功能块内其余 9 类表格 | **保留模板表头 + 示例数据**作填写指引 |
| 非功能需求 | "3 非功能需求分析"子章节 | 无内容子章节插"（待补充）" |

### 功能块表格清单（每个 capability 复制一份，全部保留）

- 功能概述表（5×2：功能编号/功能名称/变更类型/功能触发入口/功能说明）
- 逻辑流程图（1×1）/ UI模型图（1×1）
- 处理逻辑表（1×1）
- 账务处理设计（1×1）+ 账务处理下的各机制说明
- 会计核算（1×1 说明表 + 3×6 科目表：借贷方向/科目号/科目名称/金额类型/机构/利润中心）
- 凭证打印（1×1）
- 接口设计（1×1 说明表 + 4×5 接口清单：功能/接口名称/接口地址/接口变更类型/API治理）
- 数据库设计（1×1）
- 影响范围分析（1×1 说明表 + 6×6 影响表：序号/影响系统分类/影响系统名称/影响功能/具体影响/备注）

### 自动填充范围（spec 能对上的两处）

只有**功能概述表**和**处理逻辑表**两处能从 spec 自动填，因为 spec 是行为规格（capability/requirement/scenario），而企业表格多数是设计字段（接口地址/科目号/影响系统），语义鸿沟大，spec 阶段产不出。其余表格保留模板示例数据，等 plan/task 阶段或手填。

## 技术路线演变

### v3（当前采用）：deepcopy 完整功能块 + 选择性填表

- python-docx 以对象模型操作完整模板
- deepcopy 首个示例功能块（含全部表格）作为模板
- 删除 2 个示例功能块，保留功能区总览（说明段 + 总览表）
- 每个 capability：deepcopy 功能块模板 → 改 heading2 标题 → 填功能概述表 + 处理逻辑表 → 其余表格不动
- 插入功能区总览之后、"非功能需求分析"h1 之前

### v2（已废弃）：章节映射重建

删除功能区所有内容，用 heading 骨架重建。**问题：丢失了模板的全部结构化表格**。

### v1（已废弃）：扁平插入

pandoc 生成正文整段塞进模板。未复用模板章节骨架。

### demo 闭环验证

- **demo1**：pandoc + `--reference-doc` + `--toc`，markdown 章节套用模板 heading 样式、生成 TOC
- **demo2（证伪）**：正则切分 XML + 字符串拼接破坏 OOXML 结构（表格乱、正文消失）。**放弃 XML 拼接路线**
- **demo3/3b**：python-docx 对象级填充（`addprevious`/`addnext` + `deepcopy`），样式正确承接

结论：模板始终保持完整，表格结构不被破坏。pandoc 角色弱化（仅保留前置检查，实际填充由 python-docx 完成）。

## 关键实现位置（sdd_doc.py）

- 解析：`parse_proposal`（按 `##` 分块）/ `parse_specs`（解析 openspec delta）
- 模板定位：`_find_functional_region` / `find_function_block_range`
- 功能块处理：`delete_sample_function_blocks`（保留总览）/ `fill_function_block_tables`（按 h3 标题填两个表）
- 表格填充：`_fill_overview_table`（行 1/行 4）/ `_fill_logic_table`（scenario）/ `_set_cell_el_text`（多行）
- 编排：`map_to_template`

## 自定义模板的适配要求

若用户提供自己的企业模板，需满足：
- 含 heading1"功能详细分析及设计"和"非功能需求分析"（文本匹配，用于定位功能区边界）
- 功能块以 heading2 开头，含 heading3 子章节（至少"功能概述""处理逻辑"，用于自动填表识别）
- 功能概述表为 5×2 结构（行 1 功能名称、行 4 功能说明可填）
- 处理逻辑表为 1×1 单元格

否则 `map_to_template` 会报"模板未找到功能块作为模板源"等错误。
