# 中间件使用规范

> 定义团队中间件选型、配置约定和使用模式。


---

## 1. 中间件选型原则


[SOFT_RULE] 默认技术栈：

| 类别 | 推荐方案 | 备选方案 |
|------|----------|----------|
| 缓存 | Redis | Caffeine（本地） |
| 消息队列 | RocketMQ / Kafka | RabbitMQ |
| RPC | gRPC / Dubbo | Feign（HTTP） |
| 配置中心 | Nacos / Apollo | Spring Cloud Config |
| 服务注册 | Nacos / Consul | Eureka |
| 搜索引擎 | Elasticsearch | - |
| 对象存储 | MinIO / S3 | - |

## 2. 缓存规范

> 企业项目优先遵守 `common-tech.md` 中的缓存策略，本文件为通用中间件选型补充。

### 2.1 缓存策略

[HARD_RULE] 缓存使用必须：
- 明确定义缓存键格式：`{业务域}:{实体}:{标识}`，如 `user:profile:12345`
- 设置合理的 TTL，禁止永久缓存（除非静态数据）
- 缓存穿透：使用布隆过滤器或空值缓存
- 缓存击穿：使用分布式锁或互斥重建
- 缓存雪崩：TTL 添加随机偏移量

[SOFT_RULE] 缓存 TTL 参考：
- 热点数据：5-30 分钟
- 一般数据：1-24 小时
- 静态数据：24 小时 + 主动失效

### 2.2 缓存一致性

[HARD_RULE] 缓存与数据源一致性：
- 写操作：先更新数据库，再删除缓存（Cache Aside）
- 读操作：先读缓存，未命中则读数据库并回填
- 禁止先删缓存再更新数据库

### 2.3 本地缓存

[SOFT_RULE] 本地缓存使用场景：
- 配置数据（变更频率低）
- 字典数据（只读或极少变更）
- 高频访问的小数据集

[HARD_RULE] 本地缓存必须设置最大容量和过期策略。

## 3. 消息队列规范


### 3.1 Topic/Queue 命名

[HARD_RULE] Topic 命名格式：`{环境}.{业务域}.{事件类型}`，如 `prod.payment.order-created`。

### 3.2 消息设计

[HARD_RULE] 消息体必须：
- 包含消息 ID（用于去重）
- 包含消息类型（用于路由）
- 包含时间戳（用于排序和超时）
- 使用 JSON 或 Protobuf 序列化
- 不携带敏感信息

[SOFT_RULE] 消息体大小不超过 256KB，大对象使用引用（如文件 URL）。

### 3.3 消费模式

[HARD_RULE] 消息消费必须：
- 实现幂等消费（通过消息 ID 去重）
- 设置合理的重试次数和退避策略
- 死信队列处理失败消息
- 记录消费日志（成功/失败/重试）

[SOFT_RULE] 重试策略：指数退避，初始 1s，最大 60s，最多重试 5 次。

### 3.4 顺序消息

[HARD_RULE] 需要顺序保证的场景：
- 使用分区键（如订单 ID）保证同一实体消息顺序
- 消费者单线程处理同一分区
- 禁止在顺序消息消费中异步分叉

## 4. RPC 规范


### 4.1 接口定义

[HARD_RULE] RPC 接口必须：
- 使用 Protobuf / IDL 定义契约
- 接口方法命名使用动词+名词：`CreateOrder`、`GetUser`
- 请求/响应使用独立消息类型
- 定义错误码枚举

### 4.2 超时与重试

[HARD_RULE] RPC 调用必须：
- 设置连接超时（建议 3s）
- 设置请求超时（建议 10s）
- 幂等操作可重试，非幂等操作禁止重试
- 重试次数不超过 3 次

### 4.3 熔断与降级

[HARD_RULE] 外部服务调用必须：
- 实现熔断器（失败率阈值触发）
- 定义降级策略（返回默认值/缓存/错误）
- 熔断恢复后使用半开状态探测

## 5. 配置中心规范


### 5.1 配置分类

[HARD_RULE] 配置按敏感度分级：

| 级别 | 内容 | 存储 |
|------|------|------|
| 公开 | 功能开关、UI 配置 | 配置中心 |
| 内部 | 数据库地址、端口 | 配置中心（加密） |
| 敏感 | 密码、密钥、证书 | 密钥管理服务（KMS/Vault） |

### 5.2 配置命名

[HARD_RULE] 配置键命名格式：`{业务域}.{模块}.{配置项}`，如 `payment.retry.max-count`。

### 5.3 配置热更新

[SOFT_RULE] 支持热更新的配置必须：
- 实现配置变更监听
- 变更时记录日志
- 验证新配置有效性后再应用

## 6. 可观测性集成


[HARD_RULE] 所有中间件调用必须：
- 透传 traceId 实现链路追踪
- 暴露调用指标（延迟、成功率、QPS）
- 记录结构化日志

## 7. 规格注入指引


[GUIDELINE] 项目级规格引用本规范时：
- 仅注入项目实际使用的中间件规范
- 缓存策略按业务场景选择性注入
- 安全要求全量注入
