# 接口方法标准 - 通用补充

> 定义 API 版本管理、内部接口、事件接口和通用安全要求。
> 金融行业项目优先遵守 `api-standards.md`（统一报文、仅 GET/POST）。


---

## 1. 版本管理

### 1.1 版本策略

[HARD_RULE] 版本管理遵循语义化版本：
- **主版本（Breaking）**：不兼容变更，新路径 `/v2/`
- **次版本（Additive）**：新增字段/端点，向后兼容
- **补丁版本（Fix）**：Bug 修复，行为修正

### 1.2 破坏性变更

[HARD_RULE] 破坏性变更必须：
- 提前一个版本周期通知调用方
- 旧版本至少保留 6 个月
- 提供迁移指南
- 废弃端点返回 `Deprecation` 响应头

### 1.3 向后兼容规则

[HARD_RULE] 以下变更视为向后兼容：
- 新增可选请求字段
- 新增响应字段
- 新增 API 端点
- 扩展枚举值

[SOFT_RULE] 以下变更视为破坏性变更：
- 删除或重命名字段
- 修改字段类型
- 修改必填字段
- 修改错误码
- 修改端点路径

## 2. 内部接口

### 2.1 服务间通信

[HARD_RULE] 服务间通信必须：
- 使用明确的接口定义（gRPC / Feign Client / 内部 SDK）
- 设置超时和重试策略
- 实现熔断和降级
- 记录调用链路（traceId 透传）

[SOFT_RULE] 内部接口不暴露版本号，通过服务注册发现管理版本。

### 2.2 事件接口

[HARD_RULE] 事件驱动接口必须：
- 定义事件 Schema（JSON Schema / Avro / Protobuf）
- 事件包含事件类型、版本、发生时间
- 消费者必须容忍新增字段
- 事件 Schema 变更遵循向后兼容原则

## 3. 安全要求

[HARD_RULE] API 安全：
- 所有 API 必须认证（内部 API 除外）
- 使用 HTTPS，禁止 HTTP 明文传输
- 敏感操作需要额外权限校验
- 实现请求频率限制（Rate Limiting）
- 记录所有 API 调用的审计日志

## 4. 规格注入指引

[GUIDELINE] 项目级规格引用本规范时：
- 版本管理策略全量注入
- 内部接口按需注入
- 安全要求全量注入
