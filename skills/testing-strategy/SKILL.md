# 测试策略 Skill

代码含业务逻辑时应主动加载——覆盖率是基线要求（baseline 已注入）。

## 加载决策

| 任务 | Read 这个 resource |
|------|------------------|
| 测试金字塔、单元/集成/E2E 分层、覆盖率要求、TDD 实践、测试数据、性能测试 | `resources/testing-strategy.md` |

## 核心规则（无需 Read resources 也必须遵守）

- **覆盖率 ≥ 80%**——低于此值关键路径缺陷会逃逸到生产
- **关键路径必须单测覆盖**——业务核心逻辑失效的修复成本远高于测试成本
- **测试金字塔**：单元（大量）> 集成（适量）> E2E（少量）——禁止倒金字塔
- **单元测试用 AAA 模式**：Arrange / Act / Assert
- **修复 bug 必须加回归测试**——否则同样的问题会再来
- **推荐 TDD**：先写测试再写实现，强制思考边界条件

## 主动加载时机

以下场景**应主动加载**本 skill（即使用户没明说）：

- 编写业务 Service/Manager 层代码
- 编写 Controller 层（涉及业务逻辑）
- 重构现有代码
- 修复 bug（必须加回归测试）

## 何时不应使用

- 纯静态代码分析（lint/type check 不算测试）
- 性能基准测试（不属于研发基线，按需做）
