# 数据库配置规范

> 定义数据库选型、连接配置、迁移策略和数据安全要求。
> 金融行业项目优先遵守 `database-conventions.md`（5 库合并规范），本文件为通用数据库设计补充。


---

## 1. 数据库选型


[HARD_RULE] 数据库选型原则：
- 关系型：MySQL / PostgreSQL（事务型业务）
- 文档型：MongoDB（灵活 Schema 场景）
- 时序型：InfluxDB / TDengine（监控/日志数据）
- 图数据库：Neo4j（关系密集型场景）

[SOFT_RULE] 默认选型：MySQL 8.0+ 作为主数据库。

## 2. 连接配置


### 2.1 连接池

[HARD_RULE] 连接池必须配置：
- 最小空闲连接：5
- 最大连接数：根据 `(CPU核心数 × 2) + 磁盘数` 计算
- 连接超时：30s
- 空闲超时：600s
- 最大生命周期：1800s

[SOFT_RULE] 使用 HikariCP 作为默认连接池。

### 2.2 连接参数

[HARD_RULE] 数据库连接必须：
- 使用连接池，禁止每次操作新建连接
- 启用 SSL/TLS 加密
- 配置连接验证查询
- 敏感信息通过密钥管理服务获取，禁止硬编码

## 3. 表设计规范


### 3.1 通用字段

[HARD_RULE] 所有业务表必须包含：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT / UUID | 主键 |
| `created_at` | TIMESTAMP | 创建时间 |
| `updated_at` | TIMESTAMP | 更新时间 |
| `deleted_at` | TIMESTAMP NULL | 软删除时间 |

### 3.2 命名规范

[HARD_RULE] 数据库对象命名：
- 表名：小写下划线，复数形式（`users`、`order_items`）
- 列名：小写下划线（`user_id`、`created_at`）
- 索引名：`idx_{表名}_{列名}`（`idx_users_email`）
- 外键名：`fk_{表名}_{引用表名}`（`fk_orders_user_id`）
- 唯一约束：`uk_{表名}_{列名}`（`uk_users_email`）

### 3.3 索引设计

[HARD_RULE] 索引设计原则：
- 外键列必须创建索引
- 频繁查询的 WHERE/ORDER BY 列创建索引
- 联合索引遵循最左前缀原则
- 单表索引数量不超过 10 个
- 定期分析并删除未使用索引

[SOFT_RULE] 高基数列优先创建索引。

## 4. 查询规范


### 4.1 SQL 编写

[HARD_RULE] SQL 编写规范：
- 使用参数化查询，禁止字符串拼接
- 明确指定列名，禁止 `SELECT *`
- 使用 `LIMIT` 限制结果集大小
- 避免 N+1 查询，使用 JOIN 或批量查询
- 复杂查询使用 EXPLAIN 分析执行计划

### 4.2 分页查询

[HARD_RULE] 分页查询必须：
- 使用 `LIMIT/OFFSET` 或游标分页
- 大偏移量时使用游标分页（基于上次查询的最大 ID）
- 返回总记录数（用于前端分页组件）

### 4.3 事务管理

[HARD_RULE] 事务使用规范：
- 事务范围尽可能小
- 禁止在事务中调用外部服务
- 设置合理的超时时间（默认 30s）
- 明确指定隔离级别（默认 READ COMMITTED）
- 长事务使用补偿机制替代

## 5. 数据安全


### 5.1 敏感数据

[HARD_RULE] 敏感数据处理：
- 密码使用 bcrypt/argon2 哈希存储
- 身份证号、手机号等 PII 数据加密存储
- 加密密钥通过密钥管理服务获取
- 禁止在日志中记录敏感数据

### 5.2 访问控制

[HARD_RULE] 数据库访问控制：
- 应用使用最小权限账户
- 禁止应用账户拥有 DDL 权限
- 生产数据库禁止直连访问
- 所有访问通过审计日志记录

## 6. 性能优化


[SOFT_RULE] 性能优化建议：
- 读写分离：读多写少场景
- 分库分表：单表超过 1000 万行时考虑
- 查询缓存：热点查询使用 Redis 缓存
- 慢查询监控：记录执行时间超过 1s 的查询

## 7. 规格注入指引

[GUIDELINE] 项目级规格引用本规范时：
- 表设计规范全量注入
- 连接配置按项目实际数据库选择性注入
- 安全要求全量注入
