# 企业文档写作 Skill

写 README、API 文档、变更说明、内部规范时遵守本规范。目标：让任何团队成员（含新人、外包、上下游团队）能在 5 分钟内找到自己要的答案，10 分钟内能跑通第一个 demo。

## 文档结构

### README.md（项目根）
必须包含的章节，顺序固定：

1. **项目名 + 一句话定位**：`# 项目名 —— 一句话说清这是什么、给谁用`
2. **快速开始**：5 行命令内能跑起来（install / config / run / verify）
3. **核心特性**：3-5 条 bullet，每条一句话
4. **目录结构**：标注 `src/` `tests/` `docs/` 等关键目录用途
5. **配置说明**：所有环境变量表格（变量名 / 必填 / 默认 / 说明）
6. **常用命令**：`npm run dev` / `npm test` / `npm run build` 等
7. **开发指南**：分支策略、commit 规范、PR 流程链接
8. **FAQ / 故障排查**：top 5 高频问题
9. **贡献指南 + License + 联系方式**

禁止：把 README 当 changelog、把所有架构图塞进去、写超过 500 行。

### API 文档模板
每个接口必须包含：

```markdown
## POST /v1/orders

### 简介
创建订单（一句话）

### 鉴权
- 需要 Bearer Token，scope: `write:order`

### 请求
| 字段 | 类型 | 必填 | 说明 | 示例 |
| --- | --- | --- | --- | --- |
| buyerId | string | 是 | 买家 ID | `user_01H...` |
| items | Item[] | 是 | 商品列表 | 见下 |

### 响应
- 201 Created：返回 Order 对象
- 400 VALIDATION_FAILED：字段错误
- 409 IDEMPOTENCY_REPLAY：幂等命中

### 示例
\`\`\`http
POST /v1/orders
...
\`\`\`
```

### 变更说明（CHANGELOG.md）
格式遵循 Keep a Changelog：

```markdown
## [1.2.0] - 2026-06-18
### Added
- 新增 `/v1/refunds` 退款接口

### Changed
- `POST /v1/orders` 响应增加 `idempotencyKey` 字段（兼容）

### Deprecated
- `GET /v1/users?phone=` 改用 `/v1/users:lookup`，旧接口 2026-09 下线

### Fixed
- 修复高并发下订单号重复问题（issue #123）

### Security
- 升级 `jsonwebtoken` 至 9.0.0（修复 CVE-2022-23529）
```

### ADR（架构决策记录）
`docs/adr/NNNN-标题.md`，模板：

```markdown
# ADR-0001: 选择 PostgreSQL 而非 MongoDB

## 状态
Accepted (2026-06-18)

## 背景
订单数据需要强一致 + 复杂事务，团队 PG 运维经验更多...

## 决策
使用 PostgreSQL 14 作为主存储

## 替代方案
- MongoDB：放弃，事务支持弱
- MySQL：放弃，JSON 类型生态不如 PG

## 后果
- 优点：事务、JSONB、成熟生态
- 代价：水平扩展需要额外方案（Citus / 分库分表）
```

## 语言风格

### 通用原则
- **中文优先**：内部文档默认中文，对外英文文档单独维护
- **术语统一**：建一份 `docs/glossary.md`，全项目用同一译名
  - 例如 `token` 统一译"令牌"，不要混用"凭证/token/口令"
  - 例如 `release` 统一译"发布"，不要混用"发版/上线/部署"
- **祈使句**：指令性内容用祈使句（"必须..."、"禁止..."），不要"我们应该..."
- **避免口语化**：禁止"亲"、"简单来说"、"其实"、"反正"、"基本上"
- **避免歧义词**：禁止"很快"、"较少"、"较多"，必须给具体数字（"< 100ms"、"≤ 10 条"）

### 示例对比
❌ 口语化：
> 这个接口其实挺简单的，就是查一下用户信息，反正差不多就行。

✅ 规范：
> `GET /v1/users/{id}` 返回指定用户的公开信息，响应延迟 P99 < 200ms。

### 中文排版
- 中英文之间加空格：`使用 PostgreSQL 14`
- 中文与数字之间加空格：`延迟 < 200 ms`
- 全角标点用于中文，半角用于代码与英文：`返回"成功"，code = 0`
- 代码块必须标注语言：`` ```http ``、`` ```bash ``、`` ```json ``

## 维护规则

### 同步更新原则
- 代码 PR 修改了对外行为（接口、配置、CLI 参数）→ **同 PR 必须更新文档**
- 文档过时是 bug，不是"以后再说"；评审时 reviewer 必须检查
- 删除/重命名公开 API → CHANGELOG 必须有 `Deprecated` 或 `Removed` 条目

### 版本与归属
- 每份规范文档顶部必须有：`版本`、`最后更新`、`负责人`、`评审状态`
  ```markdown
  > 版本：v1.3 · 最后更新：2026-06-18 · 负责人：@zhang · 评审：Approved
  ```
- 规范文档变更必须走 PR + 至少 1 人 review，禁止直接 push

### 链接与失效检查
- 文档内引用必须用相对路径（`../adr/0001-pg.md`），禁止绝对 URL 指本仓库
- 外链必须定期检查（CI 跑 `lychee` / `markdown-link-check`），失效链接自动告警
- 截图/架构图必须存仓库 `docs/assets/`，禁止外链图床

### 反模式（禁止）
- ❌ "TODO: 待补充" 长期挂着（要么写，要么开 issue 跟踪）
- ❌ 复制粘贴官方文档原文（要么链接，要么按团队场景改写）
- ❌ 在文档里写实现细节（应放在代码注释或 ADR）
- ❌ 把临时会议纪要 / 聊天记录当文档（应整理成正式 ADR / RFC）
- ❌ 多份"事实文档"互相矛盾（README 与 docs/ 说不同的事）

## 自检清单（提交前）

- [ ] 5 分钟内新人能根据 README 跑起来
- [ ] 所有接口都有完整 API 文档
- [ ] CHANGELOG 已记录本次变更
- [ ] 中英文/数字间空格正确
- [ ] 没有口语化、歧义词、TODO 残留
- [ ] 所有链接可访问
- [ ] 代码示例可直接复制运行
